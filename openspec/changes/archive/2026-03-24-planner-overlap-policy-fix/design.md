## Context

The planner's Pass 2 scheduling currently has two reliability gaps: it uses brittle title-specific logic to preserve one schedule edge case, and it does not consistently reason over occupied time windows from pre-existing `tasks.org` content. This leads to avoidable same-slot collisions and hidden regressions when integration runs start from a non-empty task file.

This change removes title-based behavior and replaces it with policy-based handling for all pre-existing tasks. It also strengthens planning context (including priority) and extends integration coverage to assert that existing TODO entries are preserved while new tasks are scheduled against occupied windows by default.

Constraints:
- Existing Pass 1 task generation format stays unchanged.
- No calendar system integration is introduced.
- Existing pre-existing TODO metadata authoring rules are not redesigned.

## Goals / Non-Goals

**Goals:**
- Preserve pre-existing scheduled TODO timestamps exactly during Pass 2, using a generic rule rather than hardcoded title matching.
- Keep pre-existing unscheduled TODOs unscheduled.
- Make overlap avoidance with pre-existing occupied windows the default policy for newly scheduled tasks.
- Allow overlap only for explicit exceptions: preserved pre-existing schedules and high-priority urgency.
- Include priority in Pass 2 planning context when present.
- Add integration checks that verify immutability and scheduling-policy invariants with actionable diagnostics.
- Enforce explicit immutability invariants for pre-existing TODOs: no mutation, removal, reorder, or re-timestamp.

**Non-Goals:**
- Rewriting Pass 1 output format.
- Implementing deterministic slot-packing or optimization algorithms.
- Adding external calendar sync.
- Changing legacy semantics of tags beyond needed context enrichment.

## Decisions

1. Use a schedule-preservation classifier based on task state, not task title.
   - Decision: in Pass 2 input preparation, classify each task as pre-existing-scheduled, pre-existing-unscheduled, or newly-generated by parsing current `tasks.org` state before scheduling edits.
   - Rationale: title matching is fragile and does not generalize; state-based classification scales to all tasks.
   - Alternative considered: keep hardcoded title allowlist and add more names. Rejected because it is unmaintainable and still misses unseen tasks.

2. Treat pre-existing scheduled windows as immutable occupied constraints.
   - Decision: build an occupied-window set from pre-existing scheduled TODOs and inject it into Pass 2 context as non-mutable schedule context.
   - Rationale: this guarantees preservation and gives the model explicit collision context.
   - Alternative considered: post-hoc mutation rollback after LLM output. Rejected because it obscures planner intent and can create secondary inconsistencies.

3. Enrich anonymized task context with priority for Pass 2.
   - Decision: include priority field alongside existing ID/tag/schedule-state context entries.
   - Rationale: priority is required to justify overlap exceptions for urgent tasks.
   - Alternative considered: infer urgency from tags only. Rejected because priority is an explicit existing signal and should not be approximated.

4. Strengthen prompt policy to default-no-overlap with explicit exception clause.
   - Decision: update Pass 2 prompt instructions to state overlap is disallowed by default, with overlap allowed only for high-priority urgency and never by mutating preserved pre-existing schedules.
   - Rationale: policy language should encode intended behavior and reduce ambiguous planner outputs.
   - Alternative considered: leave prompt broad and enforce via hard validator only. Rejected because clearer prompting reduces invalid proposals earlier.

5. Add fixture-backed integration coverage for pre-existing `tasks.org`.
   - Decision: add/extend integration fixture via the same WebDAV-style path used by integration workflow, including mixed scheduled/unscheduled/priority/tag cases and minimum counts from proposal.
   - Rationale: regression risk is in real workflow composition, not isolated helper behavior.
   - Alternative considered: unit-only assertions. Rejected because ordering and byte-stability guarantees are integration-level properties.

6. Enforce immutability and overlap assertions with task-level diagnostics.
   - Decision: assert pre-existing TODO entries are not mutated, removed, reordered, or re-timestamped (with unscheduled entries remaining unscheduled), and assert new tasks avoid occupied windows unless exception criteria are met.
   - Rationale: failures should identify exact offending task to speed triage.
   - Alternative considered: coarse pass/fail checks. Rejected because they are hard to debug and regressions may be subtle.

## Risks / Trade-offs

- [Classification mistakes between pre-existing and newly-generated tasks] -> Mitigation: derive identity from stable task IDs/order snapshot before Pass 2 writes, and cover with integration fixtures containing mixed states.
- [LLM still proposes occasional overlaps under complex density] -> Mitigation: combine stronger prompt policy with post-response invariant checks that flag illegal overlaps.
- [Byte-stability checks become brittle if formatting side effects change] -> Mitigation: scope invariants to pre-existing entries and explicitly allow expected append-only new tasks.
- [Priority exception overused, reducing no-overlap quality] -> Mitigation: encode explicit exception threshold language and assert non-priority tasks do not collide with occupied windows.

## Migration Plan

1. Remove title-based preservation logic in Pass 2 and replace with state-based pre-existing schedule preservation.
2. Update Pass 2 context serialization to include priority and occupied-window context.
3. Update Pass 2 prompt text to default no-overlap with explicit exceptions.
4. Extend integration fixture setup to include required pre-existing `tasks.org` shape.
5. Add/adjust integration assertions for immutability and overlap exception policy.
6. Validate via test suite and integration assertion checks; if regressions occur, rollback by restoring previous Pass 2 policy logic while keeping new coverage scaffolding isolated.
